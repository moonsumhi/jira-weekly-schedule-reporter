# Development Completion Test Gate

## Purpose

모든 개발 작업이 완료된 직후, 변경사항과 관련된 테스트를 반드시 수행하여 기능 오류, 회귀 문제, 빌드 실패, 타입 오류, 권한 문제, 예외 처리 누락을 확인한다.

이 스킬은 개발 작업의 선택적 보조 절차가 아니라 **개발 완료를 판정하기 위한 필수 품질 게이트**다.

코드를 작성하거나 수정한 후 테스트를 수행하지 않은 상태에서는 작업이 완료된 것으로 간주하지 않는다.

---

## Mandatory Invocation

다음 작업 중 하나라도 수행했다면 구현 직후 반드시 이 스킬을 실행한다.

* 신규 기능 개발
* 기존 기능 수정
* 버그 수정
* 리팩터링
* API 추가 또는 변경
* 데이터베이스 스키마 변경
* 쿼리 변경
* 권한 및 인증 로직 변경
* UI/UX 변경
* 폼, 검색, 필터, 정렬, 페이징 변경
* 상태 관리 로직 변경
* 공통 컴포넌트 변경
* 설정 파일 변경
* 라이브러리 또는 의존성 변경
* 배포 설정 변경
* 테스트 코드 변경
* 성능 개선
* 보안 관련 변경

단순 문서 수정만 수행했고 실행 코드에 영향이 없는 경우에만 생략할 수 있다.

사용자가 별도로 테스트를 요청하지 않았더라도 코드 변경이 있었다면 자동으로 수행한다.

“시간이 부족하다”, “변경 범위가 작다”, “단순 수정이다”, “문법상 문제없어 보인다” 등의 이유로 테스트를 생략하지 않는다.

---

## Core Rule

다음 순서를 반드시 지킨다.

1. 변경사항 파악
2. 영향 범위 분석
3. 테스트 계획 수립
4. 관련 테스트 코드 확인
5. 필요한 테스트 추가 또는 수정
6. 정적 검사 실행
7. 단위 테스트 실행
8. 통합 테스트 실행
9. 빌드 실행
10. 실제 기능 동작 검증
11. 실패 원인 분석
12. 코드 또는 테스트 수정
13. 전체 관련 테스트 재실행
14. 결과 보고

테스트가 실패한 상태에서는 작업 완료를 선언하지 않는다.

---

## Step 1. 변경사항 분석

먼저 이번 작업에서 변경된 파일과 기능을 확인한다.

다음 항목을 정리한다.

* 변경된 파일
* 추가된 파일
* 삭제된 파일
* 변경된 함수 및 클래스
* 변경된 API
* 변경된 데이터 구조
* 변경된 화면
* 변경된 권한 로직
* 변경된 외부 연동
* 영향을 받을 수 있는 기존 기능

Git을 사용할 수 있다면 다음 정보를 활용한다.

* `git status`
* `git diff`
* `git diff --stat`
* 변경 파일 목록
* 현재 브랜치와 기준 브랜치의 차이

단순히 변경한 파일만 테스트하지 말고, 변경된 코드가 호출하거나 의존하는 영역까지 추적한다.

---

## Step 2. 프로젝트 테스트 방식 파악

테스트 명령어를 임의로 추측하지 않는다.

먼저 다음 항목을 확인한다.

* `package.json`
* `pom.xml`
* `build.gradle`
* `build.gradle.kts`
* `pytest.ini`
* `pyproject.toml`
* `tox.ini`
* `Makefile`
* CI 설정
* 프로젝트 문서
* 기존 테스트 디렉터리
* 기존 테스트 코드
* 기존 lint 및 build script

프로젝트에서 이미 사용하는 테스트 프레임워크와 명령어를 우선 사용한다.

예:

* Jest
* Vitest
* React Testing Library
* Cypress
* Playwright
* JUnit
* Mockito
* Spring Boot Test
* pytest
* unittest
* Go test
* Maven
* Gradle
* ESLint
* TypeScript compiler

프로젝트에 테스트 체계가 없다면 가능한 범위에서 최소한 다음을 수행한다.

* 컴파일 또는 빌드
* 타입 검사
* 정적 분석
* 주요 함수 직접 검증
* API 요청 검증
* 핵심 사용자 흐름 검증

---

## Step 3. 테스트 계획 수립

변경사항을 기준으로 테스트 계획을 먼저 만든다.

테스트 계획에는 최소한 다음 내용을 포함한다.

* 정상 동작
* 입력값 경계 조건
* 잘못된 입력
* 빈 값과 null
* 권한이 있는 사용자
* 권한이 없는 사용자
* 데이터가 없는 경우
* 데이터가 많은 경우
* API 실패
* 네트워크 실패
* 중복 요청
* 상태 전환
* 기존 기능 회귀 가능성

변경과 관련 없는 테스트를 무작정 대량 실행하는 것만으로 충분하다고 판단하지 않는다.

반드시 **변경 기능을 직접 검증하는 테스트**가 포함되어야 한다.

---

## Step 4. 테스트 코드 작성 및 보완

변경한 기능에 적절한 테스트가 없으면 테스트 코드를 추가한다.

기존 테스트가 있다면 변경된 요구사항과 일치하는지 검토하고 필요한 경우 수정한다.

테스트는 구현 세부사항보다는 실제 동작과 요구사항을 검증해야 한다.

각 테스트는 다음 기준을 만족해야 한다.

* 무엇을 검증하는지 명확한 이름
* 독립적으로 실행 가능
* 실행 순서에 의존하지 않음
* 테스트 데이터 정리
* 외부 시스템 의존 최소화
* 성공 조건과 실패 조건이 명확함
* 단순히 코드 실행 여부가 아니라 결과를 검증함

테스트를 통과시키기 위해 검증을 약화하거나 의미 없는 assertion을 사용하지 않는다.

금지 예:

* 항상 참인 assertion
* 오류를 무조건 무시하는 try-catch
* 실패한 테스트를 skip 처리
* 테스트 대상 로직을 mock으로 완전히 대체
* 예상 결과를 구현 결과에 맞춰 임의 변경
* 기존 테스트 삭제
* 검증 코드 제거

---

## Step 5. 필수 검사 수행

프로젝트가 지원하는 범위에서 다음 검사를 순서대로 수행한다.

### 1. 정적 검사

* lint
* format check
* type check
* import 오류
* 사용하지 않는 코드
* 문법 오류

### 2. 단위 테스트

변경된 함수, 서비스, 컴포넌트, 유틸리티와 직접 관련된 테스트를 실행한다.

### 3. 통합 테스트

다음 연결 구간을 확인한다.

* Controller → Service
* Service → Repository
* Frontend → API
* API → Database
* 인증 → 권한 검사
* 상태 변경 → 후속 처리
* 이벤트 → 알림 또는 이력 저장

### 4. 빌드

실제 배포 가능한 상태인지 확인하기 위해 전체 빌드를 실행한다.

### 5. 회귀 테스트

변경으로 영향을 받을 가능성이 있는 기존 테스트를 실행한다.

가능하다면 전체 테스트 스위트를 실행한다. 전체 테스트가 지나치게 크거나 실행 환경 제약이 있다면 관련 모듈 전체 테스트를 실행하고, 실행하지 못한 범위를 명확히 기록한다.

---

## Frontend Testing Requirements

프론트엔드 변경 시 최소한 다음을 확인한다.

* 화면 렌더링
* 로딩 상태
* 빈 데이터 상태
* 오류 상태
* 버튼 활성화 및 비활성화
* 입력값 검증
* 검색
* 필터
* 정렬
* 페이징
* 초기화
* 모달 열기와 닫기
* 저장과 취소
* 중복 클릭
* API 실패
* 권한별 화면 노출
* 새로고침 후 상태
* 상세 화면 진입 후 목록 복귀
* 작은 화면에서 레이아웃
* 키보드 Enter, Tab, Escape 동작
* 접근성 라벨

검색 및 필터 기능을 변경했다면 특히 다음을 검증한다.

* 검색어만 적용
* 단일 필터 적용
* 복수 필터 조합
* 검색어와 필터 조합
* 필터 개별 제거
* 전체 초기화
* 날짜 시작일과 종료일
* 잘못된 날짜 범위
* 검색 후 페이지 1 초기화
* URL 쿼리 파라미터 복원
* 결과 없음
* 요청 중 중복 검색 방지

가능한 경우 React Testing Library, Playwright, Cypress 등 프로젝트에서 사용하는 테스트 도구로 실제 사용자 동작을 검증한다.

---

## Backend Testing Requirements

백엔드 변경 시 최소한 다음을 확인한다.

* 정상 요청
* 필수값 누락
* 잘못된 형식
* 존재하지 않는 데이터
* 중복 데이터
* 권한 없음
* 인증되지 않은 사용자
* 상태 전환 조건
* 트랜잭션 롤백
* 예외 응답 형식
* 페이징
* 정렬
* 검색 조건 조합
* 데이터 저장 결과
* 수정 이력
* 삭제 또는 비활성화
* 동시 요청 가능성

API를 변경했다면 요청 DTO, 응답 DTO, Controller, Service, Repository, 권한 검사까지 연결해 검증한다.

HTTP 상태 코드와 응답 본문을 모두 확인한다.

---

## Authorization and Security Testing

권한 관련 코드가 변경되었거나 사용자별 데이터 노출 가능성이 있다면 다음 테스트를 반드시 수행한다.

* 관리자 접근 가능
* 일반 사용자 접근 가능 범위
* 다른 팀 사용자 접근 차단
* 요청자 접근
* 담당자 접근
* 참조자 접근
* URL 직접 접근 차단
* API 직접 호출 차단
* 프론트엔드에서 숨겨진 기능의 백엔드 차단
* 다른 사용자 ID 또는 팀 ID 조작
* 존재 여부를 노출하는 오류 메시지
* 첨부파일 직접 URL 접근
* 민감 데이터 응답 포함 여부

화면에서 버튼을 숨기는 것만으로 권한 검증이 완료됐다고 판단하지 않는다.

반드시 서버 측 권한 검증을 확인한다.

---

## Database Testing

데이터베이스 관련 변경 시 다음을 확인한다.

* 마이그레이션 성공
* 기존 데이터 호환
* null 허용 여부
* 기본값
* 인덱스
* 외래키
* 중복 제약
* 조회 성능
* 롤백 가능성
* 잘못된 데이터 입력 차단
* 트랜잭션 범위
* 삭제 시 연관 데이터 영향

운영 데이터에 직접 위험한 명령을 실행하지 않는다.

테스트 데이터베이스나 안전한 개발 환경을 사용한다.

---

## Failure Handling

테스트가 실패하면 다음 절차를 따른다.

1. 실패 로그를 읽는다.
2. 최초 실패 원인을 찾는다.
3. 구현 오류인지 테스트 오류인지 구분한다.
4. 근본 원인을 수정한다.
5. 실패한 테스트를 다시 실행한다.
6. 관련 테스트 전체를 다시 실행한다.
7. 빌드와 정적 검사를 다시 수행한다.

한 테스트를 수정한 후 해당 테스트만 통과했다고 종료하지 않는다.

변경 영향 범위의 테스트를 다시 실행하여 회귀가 없는지 확인한다.

동일한 실패가 반복되면 임시 우회 코드를 추가하지 말고 구조적 원인을 분석한다.

---

## Environment Limitations

테스트 실행이 환경 문제로 불가능한 경우에도 테스트를 생략했다고 숨기지 않는다.

다음 내용을 명확히 기록한다.

* 실행하려던 명령
* 실행하지 못한 이유
* 오류 메시지
* 필요한 환경 또는 의존성
* 실제로 확인한 범위
* 확인하지 못한 위험
* 사용자가 직접 실행할 명령

다음 표현만으로 끝내지 않는다.

* “테스트하지 못했습니다.”
* “환경이 없습니다.”
* “아마 정상입니다.”
* “코드상 문제없어 보입니다.”

실행 가능한 대체 검증이 있다면 반드시 수행한다.

---

## Completion Criteria

다음 조건을 모두 만족해야 개발 작업이 완료된 것으로 판단한다.

* 요구사항이 구현됨
* 관련 테스트가 존재함
* 관련 테스트가 실행됨
* 테스트 결과가 통과함
* lint 또는 정적 검사가 통과함
* 타입 검사가 통과함
* 빌드가 통과함
* 주요 사용자 흐름을 검증함
* 권한 변경 시 권한 테스트가 통과함
* 실패한 테스트를 숨기거나 제외하지 않음
* 실행하지 못한 검사가 명확히 기록됨

하나라도 충족되지 않으면 “완료”라고 말하지 않는다.

대신 “부분 완료”, “검증 필요”, “환경 제약으로 미확인”이라고 정확히 표시한다.

---

## Final Report Format

모든 개발 및 테스트가 끝난 후 반드시 다음 형식으로 결과를 보고한다.

### 구현 내용

* 변경한 기능
* 주요 변경 파일
* 핵심 설계 내용

### 수행한 테스트

| 구분     | 테스트 내용          | 실행 명령 | 결과    |
| ------ | --------------- | ----- | ----- |
| 정적 검사  | lint 및 코드 품질 검사 | 실제 명령 | 성공/실패 |
| 타입 검사  | 타입 오류 확인        | 실제 명령 | 성공/실패 |
| 단위 테스트 | 변경 로직 검증        | 실제 명령 | 성공/실패 |
| 통합 테스트 | 모듈 간 연동 검증      | 실제 명령 | 성공/실패 |
| 빌드     | 배포 빌드 확인        | 실제 명령 | 성공/실패 |
| 수동 검증  | 주요 사용자 흐름       | 검증 방법 | 성공/실패 |

### 검증한 시나리오

* 정상 흐름
* 예외 흐름
* 권한별 흐름
* 경계 조건
* 회귀 확인

### 발견하고 수정한 문제

* 테스트 과정에서 발견한 문제
* 원인
* 수정 내용
* 재검증 결과

발견된 문제가 없다면 “테스트 과정에서 추가로 발견된 문제 없음”이라고 적는다.

### 실행하지 못한 테스트

* 실행하지 못한 항목
* 사유
* 남은 위험
* 실행에 필요한 명령 또는 환경

모든 검사를 수행했다면 “없음”이라고 적는다.

### 최종 판정

다음 중 하나만 사용한다.

* `검증 완료`
* `부분 검증 완료`
* `검증 실패`
* `환경 제약으로 검증 미완료`

테스트가 실패했거나 주요 테스트를 실행하지 못했다면 `검증 완료`를 사용하지 않는다.

## Behavioral Enforcement

이 스킬이 실행되면 다음 행동을 금지한다.

* 테스트 없이 완료 선언
* 사용자에게 테스트 실행 여부를 다시 물어보기
* 테스트를 나중으로 미루기
* 실패 테스트 무시
* 실패 테스트 삭제 또는 skip
* 로그를 확인하지 않고 재실행만 반복
* 빌드 성공만으로 기능 테스트를 대신하기
* 단위 테스트 성공만으로 통합 검증을 생략하기
* UI 변경을 코드 리뷰만으로 검증 완료 처리하기
* 권한 처리를 화면 노출 여부만으로 검증하기
* 테스트 결과를 추측해서 보고하기

사용자가 명시적으로 테스트를 생략하라고 하더라도, 코드 변경 작업의 안정성을 위해 최소한 정적 검사, 관련 테스트, 빌드 검증을 수행한다. 실제 실행이 위험하거나 운영 환경에 영향을 주는 경우에는 안전한 테스트 방식으로 대체하고 그 사실을 보고한다.

---

## Automatic Trigger Guidance

이 스킬의 설명과 메타데이터에는 다음 의미가 드러나도록 작성한다.

> 코드 구현, 수정, 버그 해결, 리팩터링 또는 설정 변경이 끝난 직후 자동으로 사용한다. 개발 작업의 최종 단계에서 변경 영향도를 분석하고 관련 테스트, 정적 검사, 타입 검사, 빌드 및 회귀 검증을 수행한다. 테스트 없이 작업 완료를 선언하지 않는다.

스킬 이름 예시:

`development-test-gate`

스킬 설명 예시:

`개발 작업이 완료된 직후 자동으로 실행되는 필수 검증 스킬. 변경사항을 분석하고 관련 단위·통합·회귀 테스트, 정적 검사, 타입 검사 및 빌드를 수행하며, 실패 시 수정과 재검증을 반복한다.`

---

## Final Instruction

개발 구현을 마친 후 즉시 이 스킬의 절차를 수행한다.

테스트는 별도의 선택 작업이 아니라 구현 작업의 일부다.

코드 작성이 끝난 시점은 개발 완료가 아니다.

**관련 테스트가 통과하고 결과가 보고된 시점에만 개발이 완료된다.**
